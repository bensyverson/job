# Phase 2 — Markdown+YAML import, `schema` verb, project-root detection

## Context

Phase 1 landed a clean identity model: `--as <name>` gates every write; `ensureUser` lazily creates rows; reads are unattributed. See `project/2026-04-20-opus-feedback-plan.md` §"Phase 1 implementation notes" for the shape of what's now in the tree.

Phase 2 is the planning-input side: let an agent write a Markdown plan with a trailing YAML fence and ingest it with one command. It also lands two small orthogonal quality-of-life changes — `job schema` and `git`-style project-root detection for `.jobs.db` — that fit in the same session because they don't overlap Phase 1 or the import bulk.

**Outcome.**

1. `job import <file.md>` parses the first fenced `yaml` (or unlabeled) block whose root key is `tasks:`, validates the whole tree atomically, and creates every task inside a single transaction. On any validation error, nothing is written and no events fire.
2. `job schema` emits the JSON Schema for the import grammar to stdout. No identity required.
3. `resolveDBPath` walks up from cwd looking for `.jobs.db`, the way `git` finds `.git`. `--db` and `JOBS_DB` still override. (Phase 1 removed the identity env vars; `JOBS_DB` stays — it's location, not identity.)

This phase is additive only. No existing command surface changes.

## Gate before implementation

**`gopkg.in/yaml.v3` dependency.** Project CLAUDE.md says get user permission before adding a dependency. The first task of this phase is to ask — don't proceed past it. Reject alternatives that require hand-rolling a YAML parser; rolling our own is out of proportion for this feature. If the user declines yaml.v3, propose `sigs.k8s.io/yaml` (wraps `yaml.v3`; adds JSON-compat as a side benefit) as a fallback. If both are declined, stop and surface the blocker.

## Design summary

### Import grammar

```yaml
tasks:
  - title: "Required string"
    desc: "Optional free text; multi-line via YAML '|' allowed"
    labels: [optional, string, list]
    ref: optional-flat-namespace-handle
    blockedBy: [optional-list-of-ref-or-title]
    children:
      - title: ...           # recursive, same shape
```

- `title` required per task; all others optional.
- `ref` is author-chosen, flat namespace across the whole import (not per-subtree). Two tasks with the same `ref` → validation error.
- `blockedBy` entries resolve first as refs (checked against the flat namespace), then as verbatim-match against any task's `title` elsewhere in the import *or* against a pre-existing short ID in the database.
- An unresolved entry in `blockedBy` → atomic error citing the YAML path (e.g., `tasks[2].children[0].blockedBy[1]: "p2-tests" not found`).
- An ambiguous verbatim-title match (≥2 tasks share the same title in the import, and the `blockedBy` entry isn't a ref) → atomic error, same YAML-path format.
- `labels` is parsed and preserved in memory but **not persisted** in Phase 2. Phase 7 adds the `labels` table; Phase 2's import stores nothing for labels beyond round-tripping them through validation. Document this in the `job schema` output as "reserved; not yet persisted."

### Markdown fence selection

- Parse the input as raw text; scan for opening fence lines matching `^(```|~~~)(yaml|yml|\s*)$`.
- For each candidate fence, YAML-decode the body; accept the first block whose top-level mapping has a `tasks:` key.
- Ignore non-matching fences (other YAML for different purposes can live in the doc).
- If no block parses with `tasks:` root, error `no YAML \`tasks:\` block found in <file>`.

### Atomicity + ordering

- Phase A: parse + validate the full tree. No DB writes.
- Phase B: begin single SQL transaction; insert every task in pre-order DFS; generate short IDs via the existing `generateShortID(tx)` helper; resolve `blockedBy` after all inserts (so forward references work) by inserting rows into `blocks`.
- Echo created tasks in the input order with their new short IDs.
- On any error in Phase B (extremely rare — usually reflects a `generateShortID` exhaustion or disk failure), the transaction rolls back; nothing is visible.

### Flags

- `--dry-run`: run Phase A, skip Phase B. Emit placeholders `<new-1>`, `<new-2>`, … in echo order. `--format=json` on a dry-run reports `"id": null` per task and a top-level `"dry_run": true`.
- `--parent <id>`: roots of the imported tree become children of the existing task with that short ID. If the ID doesn't exist: error before Phase B.
- `--format=json` on a real run emits an array of `{"id": "<short>", "title": "<t>", "parent": "<short|null>"}` in input order.
- Import requires `--as <name>` (it's a write). Standard Phase-1 guard.

### `job schema`

- Emits JSON Schema (Draft 2020-12) describing the import grammar to stdout.
- No identity required (read-adjacent — it's pure output, no DB touch).
- Schema is authored by hand in a Go string constant (`schemaJSON`) — not generated from types, which would overfit to Go's `omitempty` quirks and underspecify the semantics we actually want users to see (e.g., `title` is required; `blockedBy` is an array of strings).
- Emitted JSON must round-trip `json.Unmarshal`. Tested explicitly.

### Project-root detection

- `resolveDBPath(dbFlag string) string` currently returns `dbFlag` if set, then `$JOBS_DB` if set, then the literal `.jobs.db`.
- New behavior when none of those are set: starting from `os.Getwd()`, walk up toward `/` looking for a regular file named `.jobs.db`. Return the first hit's absolute path. If none found, return the literal `.jobs.db` (so `job init` still works in cwd — existing behavior).
- `--db` and `JOBS_DB` short-circuit the walk exactly as today.
- Walk stops at filesystem root. Does not cross mounts (i.e., no special handling — `os.Stat` is enough; if stat fails we just keep walking up).
- **`job init` keeps using cwd.** It creates `.jobs.db` in the current directory regardless of any ancestor's `.jobs.db`. Otherwise `job init` inside a subfolder of an existing project would silently no-op. We call this out in help text / init output.

## Helpers reused

- **`generateShortID(tx)`** (`database.go:108`) — per-task inside the import transaction.
- **`runBlock`** / the `blocks` table insert pattern — import can replicate the insert inline (no cycle check required for imports, since the import graph is validated up front; but use the same `INSERT OR IGNORE` pattern for safety).
- **`recordEvent`** (`database.go`) — every created task gets a `created` event with the current actor, matching how `runAdd` records it.
- **`requireAs`** (`users.go`, added in Phase 1) — import's identity gate.
- **`expireStaleClaimsInTx`** — not needed; import doesn't operate on pre-existing claimed tasks.
- **`getTaskByShortID`** — resolves `--parent <id>`.

## Error wording (exact strings)

- **Missing YAML block:** `no YAML \`tasks:\` block found in %s`
- **YAML parse error:** `YAML parse error at %s: %s` where the first `%s` is a line number from yaml.v3's `*yaml.TypeError` or `Line` info on the node.
- **Missing title:** `%s: title is required` where `%s` is the YAML path (e.g., `tasks[1].children[0]`).
- **Duplicate ref:** `%s: ref %q is already used at %s`
- **Unresolved blockedBy:** `%s: blockedBy[%d] %q does not match any ref, imported task title, or existing task ID`
- **Ambiguous blockedBy by title:** `%s: blockedBy[%d] %q matches multiple tasks; use a ref or a short ID to disambiguate`
- **`--parent` target missing:** `parent task %q not found`
- **No `--as` on import:** same Phase-1 string: `identity required. Pass --as <name> before the verb.`

All errors are returned from Phase A (before any DB write). Tests assert the exact wording.

## Out of scope

- **`labels` table + `label` verb** — Phase 7. Import parses `labels` for forward compat but drops them on the floor.
- **`cancel` verb** — Phase 5. Import doesn't modify or create cancel events.
- **`--cascade` / variadic done** — Phase 4.
- **Help-text rewrite** — Phase 3. Import should show up in the new help text Phase 3 writes; for Phase 2, a single-line `Short` is enough.
- **Import-as-update / re-import** — not in this release. Every `job import` is an append; there is no matching pass against existing tasks. `ref`s are not persisted on task rows (the Open Questions section of the parent plan flagged this; Phase 2 confirms the "not persisted" choice).
- **Nested frontmatter or MDX-style YAML** — only triple-backtick/tilde fences are recognized.

## Risks / calls to flag

- **yaml.v3 dependency is a gate.** Don't start writing import code until it's approved.
- **`title` verbatim-match scope.** Plan scopes title-based resolution to (a) titles inside the same import, and (b) pre-existing short IDs in the DB. Not pre-existing titles in the DB. Verbatim-matching against the DB's entire title space invites silent collisions as the project grows; an `import` call wouldn't know whether "Write tests" in a new plan refers to an old task or is a new one with a shared name. Short IDs are unambiguous and the existing identifier we already teach. Flag if you want DB-title matching re-added — this note is here so the next agent doesn't "fix" it without realizing it was a deliberate narrowing of the spec's wording.
- **Project-root detection at `job init`.** Init keeps using cwd even if an ancestor has `.jobs.db`. This is tested (#24) because it's the one edge case where the user's intent could plausibly run either way. The rationale: the "git init inside a git repo" escape hatch exists for the same reason — sometimes you really do want a new one.
